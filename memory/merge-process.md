# 分支合并操作流程

当用户提出合并分支的要求时，**必须先查阅 `MERGE-PROCESS.md`** 文件（项目根目录），严格按照其中定义的七步流程执行：

1. **建立基线** — 切到目标分支，确保工作区干净，fetch 来源分支
2. **全面比对** — `git diff --stat` + `git log --oneline` 看全貌，逐文件审查 Java/资源配置/pom.xml/.gitignore，标注 🟢新增/🟡修改/🔴删除
3. **方案确认（关键铁律）** — **所有差异必须逐条向人汇报，由人做取舍决策**，Claude 可以给推荐方案但不能擅自决定。每轮提问需包含：差异类型、两方代码、冲突点、推荐方案及理由
4. **执行合并** — 等人完成所有决策后才执行 `git merge`，按已确认方案解决冲突，审查自动合并结果
5. **编译验证** — `mvn clean compile` 必须通过，失败禁止提交
6. **提交** — 按模板写 commit message（新功能/保留/修复/清理）
7. **两次检查** — 确认 commit、编译、修改范围

**Why:** 用户要求记住此流程，以后合并时先查阅内容。

**How to apply:** 当用户说「合并」「merge」「合入」等词时，先 Read `MERGE-PROCESS.md`，然后严格按流程执行，尤其是第三章「方案确认」——所有差异必须报告给人决策，不能擅自合并。
