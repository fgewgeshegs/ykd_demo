# 多轮对话上下文：从"失忆"到"记住聊过什么"

## 问题

```
你发："我叫小明，我在北京"
机器回："你好小明，北京是个好地方！"

你发："我叫什么？我在哪个城市？"
机器回："抱歉，我不知道你的名字。"  ← 失忆了
```

每次对话 LLM 只收到 `[system, user]` 两条消息，没有任何历史记录。

## 目标

```
你发："我叫小明，我在北京"
机器回："你好小明，北京是个好地方！"

你发："我叫什么？我在哪个城市？"
机器回："你叫小明，在北京。"  ← 记住了
```

---

## 一、整体思路

原来 LLM 每次收到的消息：

```json
{"messages": [
  {"role": "system", "content": "你是 Claw 助手..."},
  {"role": "user", "content": "我叫什么？"}
]}
```

改后 LLM 每次收到的消息：

```json
{"messages": [
  {"role": "system", "content": "你是 Claw 助手..."},
  {"role": "user", "content": "我叫小明，我在北京"},
  {"role": "assistant", "content": "你好小明，北京是个好地方！"},
  {"role": "user", "content": "我叫什么？"}
]}
```

本质就是**把历史消息拼回去**，这是 OpenAI 协议的 `messages` 数组本身就支持的——你的 LLM API 一直都能收，只是之前没往里填历史。

整体流程：

```
用户发消息
     ↓
ChatService
 ├─ ① contextStore.getHistory(userId)  → 拿历史
 ├─ ② llmClient.chat(text, history)    → 拼进 messages 发给 LLM
 └─ ③ contextStore.append(user + assistant) → 存本轮对话
```

## 三、逐个详解

### 1. Message — 单条消息记录

```java
// context/Message.java
public record Message(String role, String content) {}
```

就是 `{role: "user", content: "你好"}` 的 Java 表示。用 record 而不是 class——这就是一个不可变数据，record 自带 equals/hashCode/toString，最适合。

### 2. ContextStore — 接口

```java
// context/ContextStore.java
public interface ContextStore {
    List<Message> getHistory(String userId, int maxMessages);
    void append(String userId, String role, String content);
    void clear(String userId);
}
```

**为什么用接口而不是直接写死一个类**：现在用内存，以后想切 Redis 或数据库，实现类换一下就行，ChatService 一行不用改。

三个方法：
- `getHistory` — 取最近 N 条，按时间正序（最老的在前，最新的在后，正好塞进 messages 数组）
- `append` — 追一条，自动淘汰超出上限的老消息
- `clear` — 清空某个用户的全部历史

### 3. InMemoryContextStore — 内存实现

```java
// context/InMemoryContextStore.java
@Component
public class InMemoryContextStore implements ContextStore {

    private static final int MAX_MESSAGES_PER_USER = 50;

    // userId → 消息队列
    private final ConcurrentHashMap<String, Deque<Message>> store = new ConcurrentHashMap<>();
```

**数据结构选择**：

| 候选 | 为什么不用 |
|------|-----------|
| `List` | 删最早的效率 O(n) |
| `LinkedHashMap` | 需要手动维护顺序 |
| `Deque`（用 `ArrayDeque`）| `removeFirst` O(1)，`addLast` O(1)，刚好 |

`ConcurrentHashMap` 保证多线程安全，`synchronized` 锁 Deque 保证单用户内的消息顺序。并发不高的情况下这样最简单。

```java
@Override
public List<Message> getHistory(String userId, int maxMessages) {
    Deque<Message> messages = store.get(userId);
    if (messages == null) return List.of();  // 新用户，空列表

    synchronized (messages) {
        List<Message> all = new ArrayList<>(messages);
        int fromIndex = Math.max(0, all.size() - maxMessages);
        return List.copyOf(all.subList(fromIndex, all.size()));  // 最后 N 条
    }
}

@Override
public void append(String userId, String role, String content) {
    Deque<Message> messages = store.computeIfAbsent(userId, k -> new ArrayDeque<>());

    synchronized (messages) {
        messages.addLast(new Message(role, content));
        while (messages.size() > MAX_MESSAGES_PER_USER) {
            messages.removeFirst();  // 淘汰最早的
        }
    }
}
```

关键点：
- `getHistory` 返回 `List.copyOf` — 不可变副本，外面怎么改都不影响内部
- `append` 的 `computeIfAbsent` — 原子操作，首次访问自动创建 Deque
- `getHistory` 里的 `synchronized` — 和 `append` 用同一把锁，读到的是完整快照

### 4. LLMClient — 支持历史消息

改动点：`buildRequestBody` 多收一个 `history` 参数，在 system prompt 和当前消息之间插入历史。

```java
// 新方法：带历史的多轮对话
public String chat(String userId, String text, List<Message> history) {
    return callLLM(systemPrompt, text, history);
}

// 旧方法保留，内部传空 List，向后兼容
public String chat(String userId, String text) {
    return chat(userId, text, List.of());
}
```

`buildRequestBody` 的核心变化：

```java
private String buildRequestBody(String systemPrompt, String text, List<Message> history) {
    // ... 建 root, messages 数组 ...

    // ① system prompt（不变）
    messages.add({role: "system", content: systemPrompt});

    // ② 历史消息（新增）
    for (Message msg : history) {
        messages.add({role: msg.role(), content: msg.content()});
    }

    // ③ 当前用户消息（不变）
    messages.add({role: "user", content: text});

    // 最终 messages: [system, 历史1, 历史2, ..., 历史N, 当前user消息]
}
```

为什么历史消息在 system 和当前消息之间：这是 OpenAI 协议的标准顺序。LLM 按数组顺序理解对话——system 告诉它"你是谁"，history 告诉它"之前聊了什么"，最后的 user 是"现在要回什么"。

### 5. ChatService — 编排中心

```java
private static final int MAX_HISTORY = 20;

public String chat(String userId, String message) {
    // ① 取历史
    List<Message> history = contextStore.getHistory(userId, MAX_HISTORY);

    // ② 调 LLM（带历史）
    String reply = llmClient.chat(userId, message, history);
    if (reply == null || reply.isEmpty()) return null;

    // ③ 存本轮对话
    contextStore.append(userId, "user", message);
    contextStore.append(userId, "assistant", reply);

    return reply;
}
```

三步编排，顺序不能错：

1. **先取历史** — 拿到之前的对话，LLM 才能理解上下文
2. **再调 LLM** — 历史和当前消息一起发给模型
3. **最后存** — 调用成功后才存，失败不污染上下文

**为什么 MAX_HISTORY = 20（而不是更多）**：这是条数不是字数。20 条 ≈ 10 轮对话，对于微信聊天场景足够。再多会：
- 增加 LLM API 的 token 消耗（按 token 计费）
- 可能超出模型上下文窗口
- 太久远的历史对当前对话帮助不大

## 四、完整数据流（一次"我叫小明"→"我叫什么？"）

```
══════════════════════════════════════════════════════════
第一轮：用户发 "我叫小明，我在北京"
══════════════════════════════════════════════════════════

1. pollLoop: 收到消息 → WechatMessage{userId="user123", text="我叫小明..."}
     ↓
2. MessageRouter → intent=CHAT → AIChatHandler
     ↓
3. AIChatHandler.handle() → chatService.chat("user123", "我叫小明，我在北京")
     ↓
4. ChatService:
   ① contextStore.getHistory("user123", 20) → [] (新用户，空的)
   ② llmClient.chat("user123", "我叫小明...", [])
      → LLM 收到 messages: [system, user:"我叫小明..."]
      → 返回 "你好小明，北京是个好地方！"
   ③ contextStore.append("user123", "user", "我叫小明，我在北京")
   ④ contextStore.append("user123", "assistant", "你好小明，北京是个好地方！")
     ↓
5. pollLoop: reply="你好小明..." → 发给用户 ✓

InMemoryContextStore 现在存了 2 条：
  ["user": "我叫小明，我在北京",
   "assistant": "你好小明，北京是个好地方！"]

══════════════════════════════════════════════════════════
第二轮：用户发 "我叫什么？我在哪个城市？"
══════════════════════════════════════════════════════════

1. pollLoop: 收到消息 → WechatMessage{userId="user123", text="我叫什么？..."}
     ↓
2. MessageRouter → intent=CHAT → AIChatHandler
     ↓
3. ChatService:
   ① contextStore.getHistory("user123", 20)
      → ["user": "我叫小明...", "assistant": "你好小明..."]  ← 2 条历史
   ② llmClient.chat("user123", "我叫什么？...", [2条历史])
      → LLM 收到 messages:
        [
          {role: "system", content: "你是 Claw 助手..."},
          {role: "user", content: "我叫小明，我在北京"},        ← 历史
          {role: "assistant", content: "你好小明，北京是个好地方！"}, ← 历史
          {role: "user", content: "我叫什么？我在哪个城市？"}   ← 当前
        ]
      → 返回 "你叫小明，在北京。"
   ③ contextStore.append("user123", "user", "我叫什么？我在哪个城市？")
   ④ contextStore.append("user123", "assistant", "你叫小明，在北京。")
     ↓
4. pollLoop: reply → 发给用户 ✓

InMemoryContextStore 现在存了 4 条（最多 50，远没到上限）
```

## 五、为什么这样设计

**为什么 ContextStore 放独立 `context/` 包**：以后 VisionHandler 分析完图片，也可以 `contextStore.append("user", "我刚才给你看了这张图")`，让 LLM 后续对话中知道图片内容。放独立包而不是塞在 `ai/chat/` 下，是为跨 Handler 共享做准备。

**为什么在 ChatService 存而不是在 LLMClient 存**：LLMClient 的职责是"调 API"，不应该知道 ContextStore 的存在。存上下文是业务逻辑，属于 Service 层。如果 LLMClient 自己查自己存，哪天换成别的 LLM 提供商、或者让两个 LLM 共享同一份上下文，就很难改。

**为什么先调 LLM 再存**：如果 LLM 调用失败（超时、API 异常），reply 是 null，就不存。避免"用户说了话但没得到回复"的情况污染上下文——下次用户重说时，历史上不会多一条没回应的 user 消息。

**为什么历史按时间正序**：OpenAI 协议的 `messages` 数组按对话顺序排列，最早的在前面、最新的在后面。正序存储意味着取出来的 List 直接遍历塞进去就行，不用反转。

**为什么不限制 TTL 自动清理**：微信单用户消息量有限，50 条上限已控制内存。真要清理由用户自己调 API 或重启服务即可。加 TTL 清理引入定时线程，增加复杂度但没解决实际问题。
